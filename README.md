# PostgreSQL: хеш-сумма таблицы и репликация

PostgreSQL 16.

```
hash_sum_example.sql        — функция get_table_hash_sum
hash_sum_test_example.sql   — тесты
replication_example.sql     — репликация и тесты производительности
```

## Хеш-сумма

`get_table_hash_sum(table_name)` — принимает имя таблицы, возвращает `BIGINT`.

Столбцы берутся из `information_schema.columns` в алфавитном порядке и приводятся к тексту через вспомогательную `_hash_sum_col_transformation` — она учитывает тип поля, поэтому одно и то же значение в `integer` и `text` даст разный вклад в хеш. `NULL`-значения заворачиваются в `coalesce(..., 'NULL')`, чтобы не выпадали из расчёта.

В строку добавляется имя таблицы, поэтому одинаковые данные в разных таблицах дадут разные хеши.

Итоговая формула: `SUM(hashtextextended(row_string, 0)) MOD 10^18`. Сумма не зависит от порядка строк, для пустой таблицы возвращает 0. Вложенные вызовы `format` нужны из-за ограничения PostgreSQL — не более 100 аргументов на один вызов.

### Запуск

```sql
\i hash_sum_example.sql
\i hash_sum_test_example.sql
```

### Тесты

| # | Что проверяется | Результат |
|---|----------------|-----------|
| 1 | таблица с 34 типами полей, 1000 строк | хеш возвращён |
| 2 | пустая таблица | 0 |
| 3 | одинаковые данные в двух таблицах | хеши разные |
| 4 | изменение каждого поля меняет хеш | все 34 типа влияют |

## Репликация

Три таблицы: рабочая `work_table`, копия `replication.repl_work_table` (та же структура + `__unique_id`), очередь `replication.replication_queue` (id транзакции, время, имя таблицы, тип операции — I/U/D).

На `work_table` навешены три триггера. INSERT и UPDATE — `BEFORE`, DELETE — `AFTER`. В UPDATE дополнительно проверяется `OLD = NEW`, чтобы не писать в очередь холостые апдейты.

### Запуск

```sql
\i replication_example.sql
```

### Производительность (1 000 000 строк)

| операция | без триггеров | с триггерами | замедление |
|----------|--------------|--------------|------------|
| INSERT | 4.10 с | 20.67 с | ×5 |
| UPDATE | 8.10 с | 20.33 с | ×2.5 |
| DELETE | 0.77 с | 14.53 с | ×19 |

DELETE замедляется сильнее всего — сама операция быстрая, а триггер на каждую строку делает два INSERT в служебные таблицы.
